## 1. Environments Overview

The application runs across three environments:

| Environment | Purpose | Access |
|---|---|---|
| **Development** | Local developer machines | Developer only |
| **Staging** | Pre-production integration and UAT | Internal team |
| **Production** | Live system used by patients | Public (HTTPS) |

All environments share the same codebase and configuration pattern. Environment-specific values are **never** in source code — they are injected via environment variables, `appsettings.{Environment}.json` (non-sensitive only), or a secrets manager.

## 2. Phase 1 — On-Premise (University Hosting)

### 2.1 Infrastructure

```
Internet
    │  HTTPS (TLS 1.2+)
    ▼
[Windows Server 2022]
    ├── IIS Site 1: BADBIR.Api          → https://api.badbir.org  (or /api subdirectory)
    ├── IIS Site 2: BADBIR.Web          → https://patient.badbir.org
    └── SQL Server 2022 (existing)      → localhost or named instance
```

### 2.2 IIS Configuration

**BADBIR.Api (ASP.NET Core on IIS)**
- Install the ASP.NET Core Hosting Bundle for .NET 10.
- Application Pool: No Managed Code (Kestrel-in-process model).
- `web.config` generated by `dotnet publish` — do not manually edit.
- HTTPS binding with a valid TLS certificate.
- Application Pool identity: a dedicated service account with minimum SQL Server permissions.

**BADBIR.Web (Blazor Web App on IIS)**
- Static file serving for Blazor assets.
- Server-side Blazor requires persistent connections — ensure IIS WebSockets are enabled.
- Reverse proxies to the API on the same host.

### 2.3 Configuration Management (Phase 1)

| Config item | Location |
|---|---|
| Connection strings | Environment variable: `ConnectionStrings__BadbirDb` |
| JWT signing key | Environment variable: `Jwt__Key` |
| AES encryption key | Environment variable: `Encryption__Key` |
| Email SMTP credentials | Environment variable: `Email__Password` |
| Non-sensitive defaults | `appsettings.json` (committed to source) |
| Development overrides | `appsettings.Development.json` (committed, no secrets) |

### 2.4 Database — SQL Server 2022

- **Existing production database** — no structural changes permitted without DBA review.
- EF Core uses Database-First; no `dotnet ef database update` in production.
- Schema changes are applied via versioned SQL scripts stored in `legacy_reference/database_schema/`.
- The API's SQL Server login must have: `SELECT`, `INSERT`, `UPDATE` on form/patient tables. `DELETE` only for the holding account expiry job.

## 3. Phase 2 — AWS Cloud (Future)

> ⚠️ Phase 2 is not in scope for initial development. This section documents constraints that must be respected **now** to ensure a smooth migration.

### 3.1 Target Infrastructure

```
Route 53 (DNS)
    │
CloudFront (CDN + WAF)
    │
    ├── S3 (Static Blazor WASM assets, if applicable)
    └── Application Load Balancer
            │
            ├── ECS Fargate Task: BADBIR.Api     (Docker container)
            └── ECS Fargate Task: BADBIR.Web     (Docker container)

RDS: SQL Server (managed, replaces on-prem SQL Server)
AWS SES: Transactional email
AWS SNS / FCM: Push notifications
AWS Secrets Manager: All credentials
```

### 3.2 Dockerisation Requirements

Each project must produce a Docker image via a multi-stage `Dockerfile`.

**BADBIR.Api Dockerfile (example structure):**
```dockerfile
FROM mcr.microsoft.com/dotnet/sdk:10.0 AS build
WORKDIR /src
COPY ["src/BADBIR.Api/BADBIR.Api.csproj", "src/BADBIR.Api/"]
COPY ["src/BADBIR.Shared/BADBIR.Shared.csproj", "src/BADBIR.Shared/"]
RUN dotnet restore "src/BADBIR.Api/BADBIR.Api.csproj"
COPY . .
WORKDIR "/src/src/BADBIR.Api"
RUN dotnet publish -c Release -o /app/publish

FROM mcr.microsoft.com/dotnet/aspnet:10.0 AS final
WORKDIR /app
COPY --from=build /app/publish .
ENTRYPOINT ["dotnet", "BADBIR.Api.dll"]
```

**Environment variable injection in Docker (ECS):**
- AWS Secrets Manager secrets are injected as environment variables in ECS task definitions.
- Connection strings use the same environment variable names as Phase 1.
- No code changes are required for Phase 2 migration.

### 3.3 Config Portability Checklist

All of the following must be true before Phase 2 migration:
- [ ] No hardcoded file system paths in code
- [ ] No hardcoded server names or IPs
- [ ] Connection strings read from `IConfiguration` (not `ConfigurationManager`)
- [ ] All file uploads/storage use an `IFileStorageService` abstraction (local disk Phase 1, S3 Phase 2)
- [ ] Email uses an `IEmailService` abstraction (SMTP Phase 1, AWS SES Phase 2)
- [ ] Logging uses `Microsoft.Extensions.Logging` (structured output to stdout for Docker)
- [ ] All sensitive values read from environment variables / secrets manager (not `appsettings.json`)
- [ ] CORS `AllowedOrigins` reads from config (not hardcoded)
- [ ] `UseForwardedHeaders()` is configured for reverse proxy support (CloudFront/ALB)

## 4. Development Environment Setup

### 4.1 Prerequisites

| Tool | Minimum Version | Notes |
|---|---|---|
| .NET SDK | 10.0 | Check with `dotnet --version` |
| SQL Server | 2019 (local) or 2022 | Express edition acceptable for dev |
| Visual Studio / Rider | VS 2022 17.8+ or Rider 2024.1+ | With MAUI workload |
| MAUI Workload | Latest | `dotnet workload install maui` |
| Git | 2.40+ | — |

### 4.2 First-Time Setup

```bash
# 1. Clone the repository
git clone https://github.com/hsn953/BadbirPatient.git
cd BadbirPatient

# 2. Install MAUI workload (for mobile builds)
dotnet workload install maui

# 3. Restore all packages
dotnet restore BADBIR.PatientApp.slnx

# 4. Set local developer secrets (never commit these)
cd src/BADBIR.Api
dotnet user-secrets set "Jwt:Key" "your-local-dev-key-minimum-32-chars"
dotnet user-secrets set "ConnectionStrings:BadbirDb" "Server=localhost;Database=BADBIR_Dev;..."
dotnet user-secrets set "Encryption:Key" "your-local-aes-key"

# 5. Build non-MAUI projects
dotnet build src/BADBIR.Shared
dotnet build src/BADBIR.Api
dotnet build src/BADBIR.UI.Components
dotnet build src/BADBIR.Web

# 6. Run tests
dotnet test tests/BADBIR.Api.Tests

# 7. Run the API
dotnet run --project src/BADBIR.Api

# 8. Run the web frontend (separate terminal)
dotnet run --project src/BADBIR.Web
```

### 4.3 Mobile Build (Requires MAUI Workload + Android SDK)

```bash
# Android
dotnet build src/BADBIR.Mobile -f net10.0-android

# iOS (macOS only)
dotnet build src/BADBIR.Mobile -f net10.0-ios
```

## 5. Logging Strategy

### 5.1 Log Levels

| Environment | Default Log Level | DB Command Logging |
|---|---|---|
| Development | Debug | Information (EF SQL visible) |
| Staging | Information | Warning |
| Production | Warning | Warning (never log SQL in production) |

### 5.2 Structured Logging

All log statements must use structured (semantic) logging:

```csharp
// Correct
_logger.LogInformation("Patient {PatientId} submitted {FormType} at {SubmittedAt}", 
    patient.PatientId, formType, DateTime.UtcNow);

// Incorrect
_logger.LogInformation($"Patient {patient.PatientId} submitted form");
```

### 5.3 Log Output

- **Phase 1 (IIS):** File sink (rolling daily, max 10 files, 50 MB each) + Windows Event Log.
- **Phase 2 (Docker/ECS):** stdout only (consumed by CloudWatch Logs). No file sink.
- Abstraction: `ILogger<T>` from `Microsoft.Extensions.Logging` — no direct Serilog/NLog references in business code.

## 7. Security Hardening Checklist (Production)

- [ ] HTTPS only; HTTP → HTTPS redirect enforced
- [ ] HSTS header: `Strict-Transport-Security: max-age=31536000`
- [ ] Content Security Policy header configured
- [ ] JWT `Key` minimum 256 bits, rotated annually
- [ ] AES encryption key stored in secrets manager, not in code
- [ ] CORS policy lists specific origins (no wildcard)
- [ ] SQL Server account has minimum required permissions
- [ ] `appsettings.json` does not contain any credentials or signing keys
- [ ] Application runs under a non-admin service account
- [ ] IIS request filtering: max request size configured
- [ ] Dependency vulnerability scan on every build (`dotnet list package --vulnerable`)
